# Codex Goal: ChessQuest Hardening Backlog

## Purpose

This document freezes the current ChessQuest learning so the harness can keep improving without drifting into an attempt to turn an LLM into a chess master.

ChessQuest is a harness for agentic planning under a formal game, not a chess engine project and not a coaching UI.

The useful target is:

```text
Can an agent act as a chess player using public board state, legal rules, bounded self-authored projections, receipts, and role/goal context?
```

The target is not:

```text
Can the harness secretly make the model play strong chess?
```

Core invariant:

```text
The model acts.
The harness verifies.
Receipts prove mutation.
The rules engine owns legal truth.
Strategy, doctrine, and turn intent shape cognition but do not prove chess truth.
```

## Hard Boundary

Allowed in strict benchmark mode:

- public board state
- legal move affordances
- current check status
- UCI move execution
- read-only projection of agent-authored lines
- claim verification for agent-authored lines
- public phase goals from orchestration
- public operating doctrine
- neutral factual threat/attack data when the chosen surface exposes it
- deterministic receipts and phase reports

Forbidden in strict benchmark mode:

- host-selected candidate moves
- best move hints
- move rankings
- engine scores
- principal variations
- tactical labels such as "fork", "pin", "blunder", "brilliant", or "mate in N"
- hidden objective solution lines
- opponent policy internals
- recommendations such as "look at Nf3"
- board-specific tactical salience generated by the host

The host may shape protocol-level and phase-level salience.

The host must not choose board-specific tactical salience.

## Current Loss Threads

These are the known failure lines from recent runs and reviews.

### 1. Legal Was Treated As Safe

Observed behavior:

```text
The model used legal move listing and one-ply projection as if they proved the move was safe or good.
```

Example pattern:

```text
project_line accepted c3e4
agent rationale treated that as tactical safety
opponent replied and material was lost
```

Current mitigation:

- playing doctrine says legal does not mean safe
- decision protocol says one-ply projection is not safety
- cockpit warns on unsupported safety claims
- tool descriptions have been tightened

Implemented hardening:

- explicit `project_line` response flags:
  - `legalProjectionOnly: true`
  - `moveQualityKnown: false`
  - `safetyKnown: false`
  - `opponentReplyModeled: line.Length >= 2`
- tests for one-ply and agent-authored opponent-reply projection semantics

Remaining work:

- add prompt regression tests that tactical safety claims require evidence or explicit uncertainty

### 2. Threat Blindness

Observed behavior:

```text
The model misses immediate captures and opponent replies because current tools do not expose neutral threat facts.
```

Implemented state:

- the agent can see board state
- the agent can list legal moves for the current side
- the agent can project self-authored lines
- `StrictRefereeThreatAware` exposes a direct neutral list of opponent legal captures and attacked agent pieces

Impact:

The agent must infer threats from board geometry. That is exactly where LLMs are unreliable.

Implemented tool:

```text
chess.inspect_attacks
```

Preferred name:

```text
chess.inspect_attacks
```

Alternate acceptable name:

```text
chess.list_public_attacks
```

Tool role:

```text
Neutral public-rule perception primitive.
Not an evaluator.
Not a coach.
Not a best-move oracle.
```

Allowed output:

```json
{
  "agentColor": "white",
  "sideToMove": "white",
  "agentKingInCheck": false,
  "opponentLegalCaptures": [
    {
      "move": "c7f4",
      "from": "c7",
      "to": "f4",
      "capturedPiece": "white_bishop"
    }
  ],
  "attackedAgentPieces": [
    {
      "square": "f4",
      "piece": "white_bishop",
      "attackers": ["c7"]
    }
  ],
  "readOnly": true,
  "evaluation": null,
  "recommendation": null
}
```

Forbidden output:

- "dangerous"
- "hanging"
- "best response"
- "you should"
- "winning tactic"
- score/evaluation
- ranking by importance
- named tactics

Tests:

- returns opponent legal captures from current FEN
- returns attacked agent pieces
- read-only and does not mutate FEN/history
- does not expose score, best move, tactic labels, recommendation, or ranking
- active strict tool surface contains `inspect_attacks` only when the chosen surface mode allows threat perception

Resolved design question:

Should `inspect_attacks` be part of `StrictRefereeProjected` by default, or a separate `StrictRefereeThreatAware` surface level?

Decision:

It is behind a new surface level first:

```text
StrictRefereeThreatAware
```

The heuristic and agent-opponent standard-start scenarios now use the threat-aware surface; `standard_start_random` remains a projected baseline.

### 3. Material Awareness Is Too Late

Observed behavior:

```text
The phase report can show material loss after the phase, but the active runner often lacks current neutral material awareness while selecting a move.
```

Current state:

- phase report includes material delta
- cockpit can show material delta after phase completion
- active tool surface does not expose neutral material count

Possible fix:

Add either:

```text
chess.get_material
```

or include neutral material counts in `chess.get_state` at higher surface levels.

Allowed output:

```json
{
  "white": {
    "pawns": 8,
    "knights": 2,
    "bishops": 2,
    "rooks": 2,
    "queens": 1
  },
  "black": {
    "pawns": 8,
    "knights": 2,
    "bishops": 2,
    "rooks": 2,
    "queens": 1
  },
  "deltaForAgent": 0
}
```

Forbidden output:

- "white is winning"
- "black is better"
- engine score
- best conversion plan

Recommendation:

Do not add this before `inspect_attacks`.

Threat perception is the sharper missing primitive. Material counts can be added after threat-aware runs prove the need.

### 4. Stale Board Contact / Playing One Move Behind

Observed behavior:

```text
The model sometimes selected moves that were legal one turn earlier or based on stale mental board state.
```

Current mitigation:

- `play_move` can require `legalMoveObservationId`
- stale legal move observations are refused
- missing legal observation id is refused when the turn intent claims current legal list basis
- cockpit now prints refused/no-op moves honestly

Remaining work:

- require a fresh `legalMoveObservationId` for all `play_move` calls in strict benchmark mode
- allow explicit bypass only for actor probes where legal list is intentionally withheld
- add cockpit warning when `play_move` is called without a legal observation id and the strict surface normally expects one

Tests:

- strict gameplay refuses play without current legal observation id
- actor probe mode still permits no-list action validation
- stale observation refusal includes current legal moves and FEN unchanged

### 5. Invalid JSON / Truncation Recovery

Observed behavior:

```text
Planner output hit MAX_TOKENS or returned malformed JSON. Repair sometimes failed and ended the run.
```

Current mitigation:

- workflow planner has bounded repair/retry for invalid plan JSON
- task planner has deterministic fallback for recoverable invalid task graphs
- task prompt now warns against invalid acceptance kinds like `ObjectiveVerifier`

Remaining work:

- persist provider finish reason on every planner failure
- distinguish malformed JSON from truncation
- include last valid context frame and failure context in repair prompt
- record repair attempts in a divergence/diagnostic ledger once Run Continuity Spine exists
- add `--max-output-tokens` defaults that are realistic for orchestration/task planning

Tests:

- invalid refinement JSON repairs once and continues
- MAX_TOKENS finish reason is logged and exposed in run log
- repair failure does not erase original provider error
- task planner invalid acceptance kind falls back or repairs without crashing

### 6. Terminal Loss Was Classified Poorly

Observed behavior:

```text
Terminal black_win/checkmate could be reported primarily as Blocked or MaxRefinementsReached.
```

Correct behavior:

```text
If the board is terminal and winner != agentColor:
  status = Failed
  stopReason = TerminalLoss
```

Implemented hardening:

- terminal loss/draw can fail via ChessQuest completion evaluation with `TerminalLoss` / `TerminalDraw`
- ChessQuest CLI runs opt into post-batch completion evaluation so terminal game truth can stop before further refinement
- orchestration acceptance rejects terminal non-agent wins before trying to refine

Remaining work:

- do not allow max refinement/budget exhaustion to mask terminal game truth

Tests:

- terminal loss returns failed outcome with `TerminalLoss`
- terminal draw returns failed outcome with `TerminalDraw`
- terminal agent win still requires `chess.complete_objective` verifier artifact for success

### 7. Phase Drift

Observed behavior:

```text
The orchestration tier can select conversion/endgame language when the agent is materially behind or unstable.
```

Current mitigation:

- playing doctrine discourages conversion unless advantage/stability is evidence-backed
- deterministic task planner selects defense when material delta is <= -3
- phase envelope includes success signals and claim discipline

Remaining work:

- add host-side phase sanity warnings:
  - conversion selected while material delta is negative
  - tactical selected immediately after material loss without stabilization rationale
  - endgame selected only because ply is high
- add optional rejection/sanitization policy for obviously incoherent phase envelopes

Recommendation:

Warn first, reject later. The purpose is to expose orchestration reasoning quality, not overconstrain it too early.

### 8. Cockpit Trace Needs Continued Semantic Compression

Observed behavior:

```text
Raw runtime events were too noisy. Semantic turn summaries are better, but phase-level summaries still need richer expectation-vs-reality continuity.
```

Current mitigation:

- cockpit prints move attempts
- phase report prints moves, material delta, terminal state
- orchestration tier prints plan, dispatch, and phase report

Remaining work:

- add GoalSpine once Run Continuity Spine exists
- print current GoalSpine at phase boundaries
- print divergences as first-class events
- keep raw event trace available in logs but not primary console projection

### 9. Board Parsing Was Proven Only Narrowly

Observed result:

```text
Board probe passed 25/25 for occupied-square piece identity with ASCII presentation.
```

What it proves:

- model can map ASCII board coordinates to piece/color identity under that probe setup

What it does not prove:

- legal move generation
- tactical reasoning
- threat perception
- future board-state transformation
- chess competence

Remaining probes:

- action probe
- constrained action probe
- claim action probe
- tactical action probe
- threat perception probe

### 10. Opponent Quality

Current opponent options:

- random legal move opponent
- pure managed heuristic opponent
- agent-backed opponent
- possible future UCI/Stockfish

Observed issue:

```text
Random opponent is too weak to diagnose play. Heuristic opponent is useful but not calibrated. Agent-vs-agent is useful for orchestration failure surfaces, not chess strength.
```

Remaining work:

- improve heuristic opponent with simple threat awareness
- add opponent policy run logging
- add consistent opponent seed reporting
- keep Stockfish/UCI optional and explicitly out of strict benchmark first slices
- keep agent-vs-agent labeled as orchestration benchmark, not chess-strength benchmark

## Open Questions For The User

1. Should `chess.inspect_attacks` be enabled in the default strict surface, or only in a new `StrictRefereeThreatAware` surface?

2. Should neutral material count be visible to the active runner, or only phase reports and GoalSpine?

3. Should `play_move` always require `legalMoveObservationId` in gameplay, even when the agent selected a move from `project_line` rather than the legal move list?

4. How much should phase sanity policy enforce?
   - warn only
   - refuse obviously incoherent phase envelopes
   - auto-sanitize and continue

5. Should actor probes be provider-eval tools only, or should they also become part of regular regression tests with deterministic/fake planners?

6. For agent-vs-agent, should both agents share the same doctrine/protocol surface, or should the opponent agent receive a simpler one-move opponent frame?

7. Is the intended benchmark score:
   - terminal win/loss only
   - survival length
   - material delta
   - legal move rate
   - claim accuracy
   - phase goal adherence
   - some composite?

Recommendation:

Start with separate metrics and avoid one composite score until there are enough runs to know what matters.
